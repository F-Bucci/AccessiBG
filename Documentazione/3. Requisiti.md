### 1. **Introduzione**

#### 1.1 **Obiettivo**
AccessiBG si propone di essere una web app che consenta alle persone con disabilità
di ottenere informazioni sull'accessibilità delle sedi dell'Università degli Studi di Bergamo.

L'obiettivo è rappresentare fedelmente le strutture dell'università, le loro possibilità
di accesso agli edifici ed eventuali ostacoli o opportunità per le persone con disabilità
(porte tagliafuoco, ascensori, rampe, bagni per disabili, ecc.), tramite un'interfaccia web.

#### 1.2 **Scopo**
Lo scopo del sistema è supportare le persone con disabilità ed i loro eventuali accompagnatori
nella pianificazione degli spostamenti, permettendo loro di usufruire delle strutture
universitarie con maggiore serenità.

#### 1.3 **Definizioni, acronimi e abbreviazioni**
Per *sedi* si intendono le seguenti strutture dell'Università degli Studi di Bergamo:
- Dalmine (Scuola di Ingegneria)
- Sant'Agostino
- Caniana
- Pignolo
- Kilometro Rosso

#### 1.4 **Riferimenti**
- IEEE Std 830-1998 – Recommended Practice for Software Requirements Specifications

#### 1.5 **Panoramica del documento**
Questo documento descrive i requisiti software del sistema AccessiBG.
La Sezione 2 fornisce una descrizione generale del prodotto, mentre la Sezione 3
presenta i requisiti specifici del sistema.


### 2. **Descrizione generale**

#### 2.1 **Prospettiva del prodotto**
AccessiBG è una web application standalone, progettata per essere utilizzata tramite browser web.

#### 2.2 **Funzioni del prodotto**
Il sistema consente di:
- visualizzare le sedi universitarie
- consultare informazioni sull'accessibilità degli edifici
- visualizzare piani e strutture interne
- consultare immagini e informazioni sui parcheggi

#### 2.3 **Caratteristiche degli utenti**
Gli utenti del sistema includono:
- persone con disabilità
- accompagnatori
- personale universitario

#### 2.4 **Vincoli**
Il progetto software sarà sviluppato utilizzando:
- **Java** come linguaggio di programmazione
- **Vaadin** per il frontend
- **Sqlite** e **Jooq** una soluzione embedded per il layer dati
- **Eclipse** come IDE di sviluppo

#### 2.5 **Presupposti e dipendenze**
Si assume che l'utente disponga di un browser web aggiornato e di una connessione Internet.


### 3. **Requisiti specifici**
Il progetto nasce a seguito delle indicazioni fornite durante il colloquio con il docente, in cui è stato richiesto di realizzare una web app simile a piattaforme esistenti per l’accessibilità, in grado di mostrare informazioni testuali e visive sui servizi disponibili, sugli ostacoli presenti e sui percorsi accessibili all’interno delle strutture.

#### 3.1 **Requisiti funzionali**
- Consentire la selezione di una sede o struttura universitaria
- Mostrare informazioni testuali sui servizi per persone con disabilità (come bagno per disabili, banchi dediccati, ecc.)
- Evidenziare eventuali ostacoli architettonici
- Indicare ingressi accessibili e non accessibili
- Visualizzare, possobilmente, una mappa del piano con servizi, ostacoli e percorsi
- 
#### 3.2 Requisiti non funzionali
- L’interfaccia deve essere semplice, intuitiva e facilmente navigabile.
- Il caricamento delle informazioni deve essere rapido
- L’applicazione deve essere utilizzabile da browser desktop
- Il sistema deve essere progettato con separazione tra frontend, backend e database.

# **Software Quality**
Per assicurare la qualità del codice, si è cercato di seguire le convenzioni Java, tra cui l'uso di Javadoc ed il plugin SonarQube per migliorare la qualità del codice. Ci si è inoltre ispirati agli aspetti di qualità di McCall:
- Correttezza: le funzionalità che il team voleva funzionassero, sono state implementate con successo;
- Affidabilità: le funzionalità lavorano accuratamente e hanno superato i test;
- Integrità: si è cercato, per quanto possibile, di separare l'accesso ai database da quello dal layer di presentazione
- Usabilità: il codice corrente è stato progettato e scritto per permettere la popolazione immediata del database e l'apertura dell'app su localhost:8080 al primo avvio;
- Manutenibilità: il processo di refactoring è stato applicato a tutti e tre i layer del software ed è stato integrato da plugin quali SonarQube/Sonarlint per assicurare una facile manutenibilità del codice;
- Testabilità: tutti i layer sono stati facilmente testati;
- Flessibilità: il sistema si è dimostrato flessibile nell'introduzione di nuove funzionalità;
- Portabilità: il codice al momento funziona su hardware diversi tra loro, inoltre l'uso di Java è una granzia riguardo per questo parametro (write once, run everywhere)
- Riutilizzabilità: si stima il codice sia relativamente riutilizzabile
- Interoperabilità: al momento il codice riesce a interfacciarsi con Google Maps e si stima l'operabilità con altri sistemi non dovrebbe seere difficile da implementare
